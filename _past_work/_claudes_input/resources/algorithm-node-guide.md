# The Algorithm Node: Your Pre-Configured ThinkerBot
## Meet Kevin - The ThinkerBot That's Ready to Go

### What is The Algorithm Node?

The Algorithm Node is a pre-configured ChatGPT custom GPT that gives you instant access to two programming mentors:

1. **Friend Computer** - The bureaucratic, formal assistant (similar to AdminBot)
2. **Kevin** - The pragmatic developer mentor (your ready-to-use ThinkerBot!)

**Access it here:** https://chatgpt.com/g/g-68a25a78f1c48191914fa0e8535b9ec3-the-algorithm-node

### Why This Matters for CSC134 Students

Remember how in Week 9+ you transition from Flash (no code) to Thinker (can write code)? Kevin IS essentially a ThinkerBot that's already set up and ready to use! No need to configure your own - just jump in and start learning.

## The Two Personalities

### Friend Computer Mode
**Personality:** Bureaucratic, formal, slightly dystopian humor  
**Best for:** Understanding requirements, documentation, process

Example responses:
- *"Ah, Citizen, retrospectives must be filed on Form 42-C: Feelings and Lessons. Please rate your morale from 1 to 7."*
- *"Infinite loops are an act of treason against productivity. Terminate the process at once!"*

### Kevin Mode (Your ThinkerBot)
**Personality:** Pragmatic, chill, gets-things-done mentality  
**Best for:** Actual coding help, debugging, practical solutions

Example responses:
- *"First off — breathe. Second — cut the scope. We don't need a full spaceship; a skateboard will roll just fine for this sprint."*
- *"Three things for your portfolio: 1) Clean README, 2) Tests that work, 3) A demo gif. Recruiters skim, they don't read."*

## How to Use for CSC134

### Weeks 1-8 (Flash Period)
- Use The Algorithm Node sparingly
- Ask Kevin for **strategies**, not code
- Example: "Kevin, what's a good approach for organizing my restaurant functions?"

### Weeks 9+ (Thinker Period)
- Kevin can now write code for you (with intentional quirks you must fix!)
- Use for code reviews and architecture decisions
- Example: "Kevin, review my menu loop and suggest improvements"

## Sample Interactions

### Debugging Help (Kevin Mode)
```
You: "My loop runs forever, help!"
Kevin: "Alright, check three things: 1) Is your condition ever false? 2) Are you actually updating the variable in the condition? 3) Add a cout to see what's happening each iteration. Infinite loops happen to everyone - you just gotta find where you forgot to increment."
```

### Project Planning (Friend Computer Mode)
```
You: "How should I structure my adventure game?"
Friend Computer: "Citizen, proper game architecture requires Form 12-B: Scene Management Protocol. You'll need: 1) A main game loop (mandatory), 2) Scene functions (highly recommended), 3) State management (for your own safety), 4) Input validation (to prevent unauthorized data corruption). Failure to comply may result in spaghetti code. That is all."
```

### Code Review (Kevin Mode)
```
You: "Review my restaurant ordering function"
Kevin: "Cool, three quick things: 1) That nested if-else chain? Make it a switch statement. 2) You're repeating the tax calculation in three places - make that a function. 3) Your variable names like 'x' and 'temp'? Future-you will hate current-you. Name them what they actually are."
```

## Best Practices

### DO:
- Use Kevin for code reviews after you've written something
- Ask for debugging strategies, not solutions
- Request architecture advice for larger projects
- Get help with best practices and patterns

### DON'T:
- Copy Kevin's code directly (it has intentional bugs!)
- Use it to bypass learning in Weeks 1-8
- Rely on it without understanding the concepts
- Submit Kevin's code as your own without fixes

## Quick Start Guide

1. **Access**: Click the link to The Algorithm Node
2. **Choose Mode**: Start with "Hey Kevin" for practical help
3. **Be Specific**: "Kevin, help me understand why my function isn't modifying the value"
4. **Learn from Responses**: Kevin explains the WHY, not just the WHAT
5. **Apply Knowledge**: Use Kevin's guidance to fix your own code

## The Kevin Philosophy

Kevin operates on these principles:
- **Ship working code first, optimize later**
- **Simple solutions beat complex ones**
- **Read the error message - it's usually right**
- **When in doubt, add more cout statements**
- **Tests are your friend, write them**
